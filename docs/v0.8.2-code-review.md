# Moongate v0.8.2 Code Review

代码审查范围：`git diff master...HEAD`（6 commit：多脚本空格泛化、13 语言语料+识别裁判、eval 工具扩充、UI/发布流程）。方法：5 维度并行只读 review agent + 逐 finding 对抗性验证（部分验证因 API 额度中断，关键项由主线程亲自复现）。共 24 条 finding + 46 条观察。

## 已修复（本分支）

### ASR-1 [P0, bug] 拉丁词尾缀无锚定拼接破坏主流字幕
- **文件**：`Sources/MoongateCore/ASR.swift:3945` ↔ `windows/MoongateCore/Asr.cs:3754`
- **问题**：`observedLatinContinuationPrefixes`（本分支 7fb199a 新增，用于重接 whisper 子词碎片如 `op`+`in`+`ion`→`opinion`）用 `leftLower.hasSuffix(片段)` 匹配，把"任何以该片段结尾的完整词"误判为子词延续。实测 `we stop in Paris at its core a simple idea` → `we stopin Paris at its corea simple idea`（英语），`salut il`→`salutil`（法语），`drop in`/`shop in`/`develop in` 全部被粘连。whisper 的词首空格信号在 `cleanedSpeechText` 已被 trim，`joinedText` 无从区分独立词与延续。双端行为一致（parity 成立但都错）。
- **修复**：改为尾部拉丁字母游程**精确相等**匹配（`trailingLatinLetterRun == 片段`）。累积链的 allowed 集合本就包含每个累积形式作独立键（如 `seitenver`/`seitenverkle`），等值匹配仍正确串联真多段子词，对完整词 fail-safe。额外移除 `"a":["core"]` 唯一危险表项（`a` 是独立冠词、`core` 是完整词，英语 `core a` 远比意大利语被切的 `Corea` 常见）。
- **测试**：Swift `testLocalASRTimingPlannerDoesNotGlueCompleteWordsToFollowingFunctionWords` + C# 镜像；第三波合成测试的 `Corea` 断言相应调整。全 118 条 ASR 契约测试仍绿（含所有多段子词重接）。

### QCLI-1 / ASR-2 [P0→P1, parity-gap] 近空转写检测的 Cf 格式字符双端分歧
- **文件**：`windows/MoongateCore/LocalAsrConfidence.cs:350` ↔ `Sources/MoongateCore/LocalASRConfidence.swift:374`
- **问题**：Swift `looksNearEmptyTranscript` 用 `CharacterSet.controlCharacters`（Foundation 语义 = Cc+Cf，忽略阿拉伯 RLM U+200F 等不可见格式符），C# 走 `ShouldIgnoreForScriptQuality` 只忽略 `UnicodeCategory.Control`（Cc）不忽略 Format（Cf）。同一含 RLM 的单 segment 转写：mac 端判 nearEmpty→强制识别路径 throw downloadFailed / 拒绝复用；Windows 端判非空→照常出字幕。本分支新增 ar（RTL，Cf 高发）语料放大暴露面。
- **修复**：统一两端都忽略 Cc+Cf（不可见标记）。Swift 端把 `assessScriptQuality`/`isFragmentedCJKLoop`/`normalizeLoopToken` 补上 `controlCharacters` 过滤（原先内部不一致，仅 near-empty 处理了）；C# 端 `ShouldIgnoreForScriptQuality` 加 `UnicodeCategory.Format`。
- **测试**：Swift `testNearEmptyDetectionIgnoresInvisibleFormatMarks` + C# `arabicFormatMarkSummary` 断言（AsrContractsTests）。

### EVAL-01 [P0, eval-integrity] 陈旧 judge 验证已被重生成的源
- **文件**：`tools/subtitle_timing_eval/run_scorecard_baseline.py`（judge 加载）
- **问题**：`load_agent_judge` 无新鲜度校验，judge 写于旧 SRT 生成之后被重生成为不同内容时仍保持"verified pass"。实证：`tedx_gateway_poetry_hi` 的 recognition judge(17:56) 早于其 SRT(18:52)，而该 SRT 曾是逐字符天城文乱码（`ज ि स क ी`），却被记 recognition=93 verified-pass，撑进已发布的 93.0 "all_dimensions_pass: True"。runner 对 `translated.srt` 早已有同款 staleness 规则（find_translated:315），judge 却漏了。
- **修复**：`load_agent_judge` 增加可选 `source_path`，judge mtime < 源 mtime 则丢弃（打印 `[stale judge skipped]`），recognition 和 translation judge 均接入。**诚实影响**：重跑 scorecard（--acoustic）recognition verified 41→31，暴露出此前门禁部分依赖陈旧 judge；overall 均分 93.0→93.2（均分未降，是验证覆盖诚实收紧）。
- **测试**：`test_stale_recognition_judge_older_than_source_is_dropped`。

### EVAL-02 [P1, parity-gap] Python eval 解析器未跟进生产的多脚本重建（EVAL-01 的根因）
- **文件**：`tools/subtitle_timing_eval/subtitle_timing_eval/asr.py:308`
- **问题**：生产 Swift/C# 的 Devanagari/spaced-script 段词重建（commit 3e68cb3/82d8761）从未镜像到 eval 的 `_parse_whisper_cpp_token_words`，导致 eval 路径对天城文产逐字符乱码 SRT——正是 EVAL-01 里那份被陈旧 judge 放行的坏产物。生产 CLI 输出正确（已实测 `जिसकी तुलना फिर नहीं की जाएगी`）。
- **处置**：用当前生产 CLI 从 raw whisper.cpp JSON 重生成 hi SRT（内容现正确）；Python 解析器的完整镜像记为后续 eval 工具改进（不阻塞发布，因生产本身正确，且重跑走的是从 raw JSON 到生产 CLI 的路径）。见 P2 记录。

### EVAL-03 [P1, security-privacy] cookie 临时副本在中断时泄漏
- **文件**：`tools/subtitle_timing_eval/run_new_corpus_e2e.sh:41`
- **问题**：`TMPC=$(mktemp); cp "$COOKIE" "$TMPC"` 把认证 YouTube cookie 复制进 $TMPDIR，清理仅靠内联 `rm -f`，脚本 `set -uo pipefail` 无 `-e` 也无 trap，任何 SIGINT/SIGTERM/中途失败都会把认证 cookie 留在临时目录。
- **修复**：加 `trap cleanup EXIT INT TERM`，用 `TMP_COOKIE` 变量登记，中断也清除。

### EVAL-06 [P2, eval-integrity] judge 分数未做范围校验

- **修复**：`_judge_score` clamp 到 [0,100]，防手写 judge 打错（880 当 88）虚高聚合。测试 `test_judge_score_is_clamped_to_valid_range`。

### EVAL-05 [P2, eval-integrity] 覆盖门用 floor 低于文档 60%

### EVAL-07 [P2, eval-integrity] 翻译 judge 仍接受 pass-only 回退
