# v0.8 Code Review（分支 `codex/v0.8-local-asr`）

> 日期：2026-06-22。范围：v0.8 相对 master 的改动（含工作树未提交部分），重点 ASR/whisper、字幕时序、队列、跨平台 parity、安全、架构、性能回退。
> 方法：5 个子系统并行深读 → 6 维度审查，每条发现由 2 个对抗验证者独立复核 → 关键承重项由我在源码 + `whisper-cli --help` 上亲自核验。
> 结论计数：27 条发现，23 条经验证确认，4 条被否定（见 §7）。

---

## 0. 总体判断

- **识别质量方向是对的**，但 whisper **时序**有 2 个 critical 根因导致出现晚、消失突兀（详见 §1，已另立 ExecPlan `docs/exec-plans/2026-06-22-whisper-subtitle-timing.md`）。
- **跨平台 parity 有真实缺口**：Windows 只暴露 3/9 个 whisper 模型（用户可感知），且时序逻辑两端手工复制无契约保护。
- **安全面基本干净**：模型/runtime 有 SHA-256 校验、HTTPS、显式用户授权执行；只有 1 个 low（符号链接未解析）值得顺手加固。
- **有 1 个 high 正确性 bug**（cue 重叠）和几个 low bug，修复都很小。

---

## 1. Whisper 字幕时序（最高优先级，headline 问题）

> 完整根因 + 优化算法见独立 ExecPlan。这里只列审查发现。

| ID | 级别 | 位置 | 问题 | 影响 | 修复方向 |
|----|------|------|------|------|----------|
| TIMING-1 | **critical** | `ASR.swift:1230-1246`（设于 1073） | `vadEnabled=true` 透传但 `WhisperCppCommandPlan` 从不发 `--vad`；argv 无 VAD/DTW/max-len | 段边缘不贴语音，onset/offset 漂移的总根源 | 接通 `--vad`（需模型，推迟）/ 先上 `-dtw` |
| TIMING-2 | **critical** | `ASR.swift:1230-1246` / 解析 1468-1484 | 无 `-dtw`，token 偏移是粗糙帧量化估计，且作为唯一真值 | 出现/消失误差原样穿透 | 加 `-dtw <点号preset>` **+ `-nfa`**（实测：默认 flash_attn 会禁用 DTW），且 parser 须改读 `t_dtw`(×10ms,-1 回退 `offsets`)——见 ExecPlan §6 |
| TIMING-5 | high | `ASR.swift:809` | `cue.start=首词偏移`，无 lead-in/回拉 | 字幕稳定**偏晚出现**、抖动 | 出现端 `leadIn≈0.12s` 前拉，单调不晚于 W |
| TIMING-3 | high | `ASR.swift:685-686, 810` | 固定 tail（0.45/0.2s）保持，与真实语音结束脱钩 | 短词过保持/长词欠保持 | tail 保留，但改"保持到下一真实 onset 前" |
| TIMING-7 | medium | `ASR.swift:708-826` | planner 内无静音吸附 / no-speech 门控 | 边界从不对音频校正 | retimer 用 next-onset / 停顿 gap 校正 |
| TIMING-6 | medium | `ASR.swift:1424-1435, 1486-1497` | token 稀疏段回退为"整段塌成一个 cue"覆盖整段 offsets | 粗糙起止 | 回退路径也走 retimer，或细分 |
| TIMING-8 | low | `ASR.swift:1230-1246` | 无 `-ml/--split-on-word`，欠标点语言撑到上限 | 大滞留、乱断 | 用 planner 内新增 `dtwPauseBreak` 替代（不改 segment 文本粒度） |

**关键结论**：旧自动字幕能到 ~90% 是因为它直接透传 YouTube **人类对齐的逐词时间戳**作锚点（`Translator.swift:1369-1429` 的 `sourceAnchoredPieces`）；whisper 没有这个外部时钟，且走的是另一套 planner + 无锚点纯 SRT，所以**结构上无法复用**——必须单独写时序算法。

## 2. 正确性 Bug

| ID | 级别 | 位置 | 问题 | 影响 |
|----|------|------|------|------|
| BUG-1 | **high** | `ASR.swift:814-818` | `end=max(end,start+minimumCueSeconds)`（818）跑在 `min(end,nextStart)`（814-815）**之后**，短 cue 被顶出去 | **相邻 cue 重叠**，渲染叠字/闪烁。✅ 我已亲自核验钳序 |
| BUG-5 | medium | `QueueManager.swift:1213-1224, 1384-1399` | 完成项本地 ASR 重跑用 `languageCode="auto"`，但产物名是 `.local-asr.<detected>.srt`，导致已存在的 SRT 不被识别为已有 artifact | 重复跑 whisper，浪费时间/电；与 §6 性能目标冲突 |
| BUG-2 | medium | `ASR.swift:1304-1320, 1371-1388` | 进度解析把含 "N%" 的**转写文本** stdout 当进度更新 | 进度条乱跳（如台词里出现 "100%"） |

> BUG-1/BUG-4/BUG-2 都在本计划 retimer 改造中顺手修掉；BUG-5 是独立小修（重跑前按 detected 语言查 SRT，或重跑统一改写固定语言后缀）。

## 3. 跨平台 Parity

| ID | 级别 | 位置 | 问题 |
|----|------|------|------|
| PARITY-1 | **high** | `Asr.cs:207-248` vs `ASR.swift:271-371` | **Windows 只暴露 3/9 个 whisper 模型**（仅 tiny/base/small-q5_1）。✅ 我已核验：Swift 9 个、Windows 3 个。Windows 用户拿不到 small.en/medium/large-v3-turbo 等，识别质量与 macOS 不对等 |
| ARCH-3 | high | `Asr.cs:462-660` vs `ASR.swift:683-862` | whisper planner 在两端**手工复制**，常量/逻辑/fixture 全靠人肉同步，无 parity 强制 |
| PARITY-2 | low | `Asr.cs:847-856` vs `ASR.swift:1094-1106` | transcript cache key 的 `audioSeed` 两端 mtime 编码不兼容（缓存不跨端复用，且并行 QA 易误判） |
| PARITY-3 | nit | `SubtitleTimingPlanner.cs:47-53` vs `.swift:41-45` | `WordTokens` 分词：C# `\p{Nd}` vs Swift Foundation alphanumerics，在非十进制字母/数字字符上结果不同 |

> 建议：补齐 Windows 9 模型（小改，本计划 M3 一并做）；时序常量集中到共享文档 + 跨端 fixture 字节对比，把 parity 从"巧合"变"结构"。

## 4. 安全与隐私（基本干净）

| ID | 级别 | 结论 |
|----|------|------|
| PATH-TRAVERSAL-1 | low | runtime bundle 相对路径在字符串层挡了 `../`，但**未对 root 做符号链接解析**（`ASR.swift:541-554`）。残留风险低，建议 `resolvingSymlinksInPath` 后校验仍在 root 内 |
| HASH-VERIFY-1 | ✅ clean | 模型/runtime SHA-256 校验存在且顺序正确（先校验后使用） |
| INSECURE-TRANSPORT-1 | ✅ clean | 下载走 HTTPS，攻击者提供的 downloadURL 触达不到下载客户端 |
| CMD-INJECT-1 | ✅ clean | whisper argv 用数组传参（非 shell），prompt/路径无命令注入 |
| UNTRUSTED-BINARY-1 | ✅ clean | 执行的 runtime 由显式用户开关 + 可执行/SHA 校验 + 文件名白名单门控 |
| LOG-LEAK-1 | ✅ clean | stderr/错误信息含本地路径但无 secret/token；16KiB 上限兜底 |

## 7. 被否定的发现（透明记录）

对抗验证否掉 4 条，避免误导：
- **TIMING-4（否）**：声称"短句滞留到 9s"——错。`min(end, start+cap)` 是天花板只缩短不拉长；真正的消失问题是 TIMING-3/根因 #4 的 nextStart 夹断，不是滞留。
- **ARCH-5（否）**：声称 planner 无法单测——错。`LocalASRSubtitleTimingPlanner` 是 `internal`，测试用 `@testable import` 即可直接调 `planCues`，`SubtitleCueSourceFragment` 是 public。
- **LOG-LEAK-1 / UNTRUSTED-BINARY-1（否为"缺陷"）**：技术描述准确，但本身是"已干净"的观察项，无可修缺陷。

## 8. 修复优先级建议

1. **先做 whisper 时序**（TIMING-1/2/3/5 + BUG-1/4/2）：用户最痛，已有独立 ExecPlan，retimer 改造一并修 bug。
2. **补齐 Windows 9 模型（PARITY-1）**：小改、用户可感知质量对等。
3. **BUG-5 重跑去重**：省时省电。
4. **抽 `WhisperCueRetimer` 独立文件（ARCH-1/2）+ 共享常量文档**：落实"whisper 单独算法"并锁 parity。
5. 顺手：PATH-TRAVERSAL-1 符号链接解析、ARCH-4 统一 `endsSentence`、PARITY-2/3。
